Technical build guide

Building steam-mate

A Tauri 2 + Svelte desktop application for Steam account switching and Yu-Gi-Oh! Master Duel asset linking

Version 0.1  ·  Windows-first  ·  target binary < 6 MB

Abstract. steam-mate is a small cross-platform desktop app, built with Tauri 2 and SvelteKit, that does two things: switch between remembered Steam accounts, and link Master Duel's per-account asset folders to one shared cache. This guide sets out the whole v0.1 build as twelve ordered steps, then reference material — the command surface, the common pitfalls, and the project's scope. The first release targets Windows; the design keeps the Steam feature cross-platform for later.

The two features
Steam account switching (cross-platform design)
Watt-Toolkit-style: close Steam, patch loginusers.vdf and Steam's internal "registry," then relaunch signed in as another account.
Master Duel asset linking (Windows-only)
Replace each per-account LocalData\<id>\0000\ with an NTFS junction to one shared LocalData\DATA\0000\ — a ~13 GB cache reused across every Konami account, rather than re-downloaded per account.

1Scaffold the application and toolchain

Start from a clean Tauri 2 + Svelte + TypeScript scaffold managed by pnpm.

Note — Commit Cargo.lock. steam-mate is a binary, not a library, so the lockfile is committed to pin exact dependency versions for reproducible builds.

Done when — the scaffold runs: pnpm tauri dev opens the window and cargo check is clean.

2Dependencies and the error model

Pin the crate versions, then define one crate-wide error type that every command returns.

Why junctions — std::os::windows::fs::symlink_dir requires administrator rights or Developer Mode; junctions do not. That settles the choice for a tool ordinary users run — needing admin rights would block many of them. The same-volume limit on junctions does not matter here, because Master Duel and its cache sit on the same drive.

A single error enum, tagged for the frontend

The #[serde(tag="kind", content="msg")] attribute serializes each variant to { "kind": "VdfParse", "msg": "…" }, which the TypeScript side pattern-matches on kind. New variants are added per feature; per-module error enums are not introduced.

Done when — AppError and AppResult compile and the dependencies resolve (cargo check).

3Project module layout

A flat, mechanical layout: each feature owns its files, and each backend module owns its commands.rs.

Settled design decisions

  • Steam is top-level, not a "game." It is the platform; account switching is a platform feature.
  • No shared trait across games. A helper moves to a shared module only once two callers need the same thing — for example, a path-joining helper stays inside steam/ until master_duel needs it too.
  • Each module owns its commands.rs. Tauri registers commands statically in main.rs; per-module files keep that mechanical.
  • No core/ until needed. Premature shared modules end up coupled.

Cross-platform configuration

  • Module level: #[cfg(windows)] pub mod master_duel; — non-Windows builds drop it entirely.
  • Function level: per-OS branches are split into sub-files (registry/{windows,linux,macos}.rs) re-exported from mod.rs.

Project constants

The fixed values referenced throughout — one authoritative place for the literal paths, keys, and magic numbers.

Done when — every file in the tree exists (stubs are fine) and cargo check passes.

4Steam — discovery and account listing

Locate Steam, then parse loginusers.vdf into typed accounts.

Caution — registry.vdf is a Valve-format text file Steam uses on Linux and macOS as the cross-platform equivalent. It is not the Windows registry; those systems have no Windows registry to write to.

Every value is a string even when it is really an integer or boolean, so each is parsed on its own. A SteamAccount is built per block and returned by steam_list_accounts; a missing file maps to SteamNotInstalled rather than a raw input/output (IO) error. If a malformed file holds a duplicate AccountName, the block with the highest Timestamp wins.

Done when — steam_list_accounts returns the real accounts from loginusers.vdf, and a missing file gives SteamNotInstalled.

5Steam — the account-switch sequence

The core routine: switch.rs runs six steps, in order, across the vdf, registry, and process modules.

  1. Read loginusers.vdf.
  2. Locate the target by AccountName, else return AccountNotFound. The rewritten text is built up front, so an unknown account fails before Steam is touched.
  3. Close Steam cleanly (on Windows, Steam.exe -shutdown), polling up to 3000 ms and force-closing any process still alive. Running it again changes nothing — it is idempotent — so an already-closed Steam returns Ok.
  4. Point auto-login at the target (the three keys listed below).
  5. Rewrite loginusers.vdf: the target receives MostRecent=1 and RememberPassword=1 (plus the offline flags when requested), and every other user receives MostRecent=0. The file is written atomically (temporary file, then rename).
  6. Relaunch Steam.

Never touch — the ssfn* files (<install>\ssfn*, or ~/.steam/ssfn*) are the cached credentials; Steam manages them. If the target account has no ssfn, the switch correctly lands on the login screen with the username pre-filled. That is expected behavior, not an error.

Caution — corrupting loginusers.vdf loses the user's entire account list. Always write to a sibling temporary file and then rename over the target.

Offline mode and threading

Passing offlineMode: true additionally sets WantsOfflineMode=1 and SkipOfflineModeWarning=1 on the target only; persona state is a v0.2 argument and is ignored. Commands are async fn, but the switch performs blocking IO and a roughly three-second kill-poll, so it runs under spawn_blocking to avoid stalling the runtime. There is no global state — every command re-reads the filesystem.

Done when — a fixture round-trips — target MostRecent=1 / RememberPassword=1, others 0, written atomically — cargo test steam::vdf is green, and on a real machine the switch closes Steam and relaunches it signed into the target.

6Steam — the account-switching page

The /steam route: a list of accounts that one clicks to switch.

  • One row per account, showing avatar, AccountName, PersonaName, and an active indicator on the MostRecent account.
  • A switch control on each non-active row invokes steam_switch_account, with a toast reporting that Steam is restarting.
  • A toolbar offers kill Steam, start Steam, clear auto-login, and an offline mode checkbox that accompanies the next switch.
  • Rows with no cached session indicate that Steam will prompt for a password.

Frontend rule — components never call invoke directly; only the wrappers in src/lib/api/*.ts do.

Done when — /steam lists the accounts with the active indicator, and clicking a non-active row switches via steam_switch_account.

7Master Duel — discovery and account data

Discover the game through Steam's library folders, then cross-reference its account folders with accounts.csv.

Install discovery

  1. Read <steam>\steamapps\libraryfolders.vdf for all library locations.
  2. For each, test for <library>\steamapps\common\Yu-Gi-Oh!  Master Duel\masterduel.exe.
  3. The first match wins; none gives GameNotInstalled("master_duel").

Caution — there are two literal spaces between Yu-Gi-Oh! and Master. Konami shipped it that way; the whitespace must not be normalized.

Each per-account 0000\ is structurally identical to DATA\0000\ — merely the 257 directories when empty. Master Duel neither detects nor cares about the junction.

md_list_accounts cross-references the LocalData\ subdirectories (excluding DATA) with the CSV. Folders absent from the CSV return empty fields, and the interface prompts for a name. The account_type is main, sub, or empty; the interface may warn before linking a main.

Done when — md_list_accounts returns every LocalData\ folder except DATA, cross-referenced with accounts.csv.

8Master Duel — link, unlink, delete, export

Point a per-account folder at the shared cache or restore it; remove an account outright; and export the account list to keep or re-import.

Critical — on unlink, use remove_dir, not remove_dir_all. The latter walks through the junction and deletes the target's contents — the shared DATA\0000\ — destroying roughly 13 GB.

Critical — force=true deletes a non-empty 0000\ before linking. It defaults to false, and the frontend must opt in explicitly, once per click.

Two helpers support the page: md_is_linked(folder_id) wraps junction::exists, returning false (not an error) for a missing path; and md_is_running() tests for masterduel.exe in the process list, matched case-insensitively.

Never touch — LocalSave\ holds the player's save data; linking and unlinking never go near it. The lone exception is an explicit account delete with deleteSaves=true (below).

Deleting an account

Deletion removes a per-account folder outright — and, on explicit opt-in, that account's saves and its row in accounts.csv. The hazard is the 0000 junction: the per-account folder may still hold a link into the shared cache, so the link is removed first, leaving a junction-free folder that a recursive delete can clear safely.

Critical — remove the inner junction with remove_dir before remove_dir_all on the account folder (steps 4 → 5). Reversing them lets the recursive delete follow the 0000 link into DATA\0000\ and wipe the shared cache.

Caution — deleteSaves=true removes LocalSave\<id> — the decks and collection for that local profile — and it can't be undone. Default it false, take a per-delete confirmation, and offer Export first.

Exporting account info

Export gathers the account list — folder id, name, type, and link state — and serializes it so the user can keep a copy or re-import it on another machine. It is read-only and never alters the install.

Note — the page offers Export as the safe step before a destructive delete, and the same JSON is the basis for the planned account-import feature.

Done when — each command matches its listing on a tempdir layout — including the unlink and delete junction-safety paths — and the Master Duel integration test is green.

9Master Duel — the asset-linking page

The /games/master-duel route: a per-account table of link toggles.

  • One row per LocalData\ folder: identifier, an editable name, a main/sub type selector, and a link toggle.
  • Inline name and type editing invoke md_save_metadata, which upserts accounts.csv.
  • Toggling a non-empty folder on prompts for the force confirmation before replacing it.
  • A shared-cache summary (path and size) and a legend for linked, normal, main, and sub states.
  • A per-row Delete action: it confirms, offers Export first, then calls md_delete_account — with a separate, clearly-marked checkbox to also remove the account's saves (deleteSaves, default off).
  • An Export button that calls md_export_accounts and saves the returned JSON through a file dialog.

Running guard — while masterduel.exe is running, every toggle is disabled and a banner is shown; any link, unlink, or delete attempt fails with GameRunning, surfaced as an error toast.

Done when — /games/master-duel shows the per-account table with link toggle, inline edit, delete (confirm + saves opt-in), and export, and the running guard blocks edits while masterduel.exe is up.

10The command surface and types

All commands are async fn returning AppResult<T>, registered in main.rs.

Caution — tauri::generate_handler! does not accept #[cfg] per item. The Windows-only Master Duel commands are gated with two conditionally-compiled Builder branches (or a helper macro); confirm the current Tauri 2 best practice.

Arg casing — the JS side passes camelCase keys (accountName, folderId, deleteSaves); the commands carry #[tauri::command(rename_all = "camelCase")] so they map onto the snake_case parameters above.

Done when — every command in the table is registered in main.rs and reachable through its api/*.ts wrapper.

11Testing

Three layers: inline unit tests, tempdir integration tests, and a manual end-to-end pass.

  • Unit. #[cfg(test)] mod tests inline per module, loading the fixtures above.
  • Integration. integration_steam_switch.rs and integration_md_link.rs against tempfile::tempdir().
  • Manual end-to-end. The checklist below, on a real Windows 11 install.

Never — run tests against the real Steam or Master Duel install. Test state is always built fresh in a tempfile::tempdir().

Manual end-to-end checklist

  1. Open steam-mate; /steam shows the real accounts.
  2. Click an account; Steam closes, then relaunches signed in to it.
  3. Clear auto-login and switch again; Steam relaunches at the login screen with the username pre-filled.
  4. Open /games/master-duel; all Konami folders and CSV content appear.
  5. Toggle the link on a non-main account; Explorer shows 0000 as a junction to DATA.
  6. Toggle while Master Duel runs; an error toast reports the game is running. Close it; the toggle works.
  7. Untoggle; the folder is a normal directory again, with 256 buckets and root.

Done when — cargo test passes (unit + tempdir integration) and the manual checklist passes on a real Windows 11 install.

12Build and distribution

Produce the release artifact and confirm the size budget.

With the release-profile settings shown earlier (opt-level="z", link-time optimization, panic="abort", and strip), the stripped Windows binary should come in under 6 MB. Going over that budget counts as a regression to investigate.

Done when — pnpm tauri build produces a stripped binary under 6 MB plus the .msi installer.

ACritical gotchas

The combined "do not forget" list.

  1. generate_handler! cannot take #[cfg] per item — use conditional Builder branches.
  2. Write loginusers.vdf and registry.vdf atomically — temporary file, then rename.
  3. steamId64 is a string in TypeScript, never a number.
  4. force=true on link silently destroys 13 GB if misused — default it false.
  5. remove_dir versus remove_dir_all on unlink — the wrong one erases the shared cache.
  6. On delete, unlink the inner junction before remove_dir_all on the account folder — otherwise the recursive delete follows the 0000 link into the shared cache.
  7. There are two spaces in Yu-Gi-Oh! Master Duel — do not normalize them.
  8. Never touch ssfn* — those are the credentials.
  9. Never touch LocalSave\ in link/unlink — only an explicit account delete with deleteSaves may remove it.
  10. registry.vdf is not the Windows registry, despite the name.
  11. Never test against the real install — always use tempdir().
  12. md_is_linked on a missing path returns false, not an error.
  13. VDF values are strings even when semantically int/bool — parse explicitly.
  14. A duplicate AccountName in the VDF resolves to the highest Timestamp.

BScope and roadmap

Out of scope — permanently

The following overlap with Watt Toolkit but are left out on purpose, so the app keeps to two features done in full instead of twenty partial ones:

  • A time-based one-time-password (TOTP) authenticator (Steam Guard)
  • Network accelerator or proxy
  • Game store or library browsing
  • Trade and market features
  • Inventory and screenshot management

Roadmap — beyond v0.1

v0.2. Linux and macOS Steam support, registry.vdf patching, persona state, avatar fetching, a "currently signed in as" indicator, a tray quick-switch, a Master Duel filesystem watcher, disk-space estimates, bulk link and unlink, a settings page, and a continuous-integration (CI) build matrix.

v0.3. A Family Share manager, avatar-.ico shortcuts, support for platforms beyond Steam, Master Duel account import (export already ships on the asset-linking page), and direct loginusers.vdf management.

Stretch. A command-line mode, a headless tray daemon, an auto-updater, and further game modules.

Localization. A future, not-yet-scheduled feature: a localized interface in English, 日本語, and 简体中文 — Master Duel's player base skews Japanese and Chinese, so it would widen reach.

steam-mate — two features, in full. This guide is one self-contained file and needs no network.
